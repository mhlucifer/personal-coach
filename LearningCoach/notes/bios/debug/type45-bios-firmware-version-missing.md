# SMBIOS Type45 missing BIOS firmware version entry

Date: 2026-05-25

Status: verified fixed. The final generated token value had been `FIRMWARE_INVENTORY_INFO = 0`, so the BIOS Firmware Type45 static record was compiled out. Enabling it made the BIOS Firmware Type45 record appear.

## Symptom

`dmidecode -t 45` shows Type45 Firmware Inventory records for:

- `BMC Firmware`, with version such as `0.13`
- `TPM Firmware`, with version such as `70039.24078912`

But there is no visible `BIOS Firmware` Type45 record / BIOS version information.

## Verification Result

After enabling the BIOS Type45 static record, `dmidecode -t 45` shows three Firmware Inventory records:

- `BIOS Firmware`
- `BMC Firmware`
- `TPM Firmware`

The verified BIOS Type45 record includes:

```text
Firmware Component Name: BIOS Firmware
Firmware Version: 0.01.01
Release Date: 05/25/2026
Lowest Supported Firmware Version: 0.01.01
Image Size: 50 MB
State: Enabled
Associated Components: 1
  0x0000
```

This confirms the root cause was the static BIOS Type45 record being disabled, not a `dmidecode` parsing issue and not a BMC/TPM dynamic Type45 issue.

## Current Best Guess

Confirmed cause:

> The BIOS Type45 static record is being compiled out because the platform override sets `FIRMWARE_INVENTORY_INFO` to `0`, which makes `TYPE45_STRUCTURE` evaluate to disabled for the static SMBIOS table.

This matches the observed result: BMC and TPM Type45 records still appear because they are generated by their own dynamic drivers, independent of the AMI static BIOS Type45 table. The missing part is specifically the static `BIOS Firmware` entry.

## How We Located It Quickly

The key was to avoid treating "Type45 not showing BIOS version" as a generic SMBIOS parsing problem. The screenshot already gave a strong clue:

- `dmidecode -t 45` showed `BMC Firmware`.
- `dmidecode -t 45` showed `TPM Firmware`.
- It did not show `BIOS Firmware`.

That means Type45 itself was not globally broken. At least two Type45 producers were working. So the useful question became:

> Which code path produces each Type45 record?

The search then split Type45 into three producers:

1. BMC Type45: dynamic record from `AmiIpmi2Pkg/Ipmi2/IpmiInitialize/IpmiSmbiosTypes.c`
2. TPM Type45: dynamic record from `AmiModulePkg/TCG2/Common/TpmSmbios/TpmSmbiosDxe.c`
3. BIOS Type45: static SMBIOS table from `AmiCompatibilityPkg/Smbios/SmbiosBoard/SmbiosStaticDataDt/SmbiosStaticData.dt`

After that split, the bug shape became clear:

- BMC/TPM records appearing proves the dynamic paths are alive.
- BIOS record missing points at the static SMBIOS table path.
- The static BIOS Type45 block is guarded by `#if TYPE45_STRUCTURE`.
- `TYPE45_STRUCTURE` comes from `FIRMWARE_INVENTORY_INFO`.
- The default SDL says `FIRMWARE_INVENTORY_INFO = 1`.
- The platform override says `FIRMWARE_INVENTORY_INFO = 0`.

So the fast path was:

```text
Runtime symptom
  -> distinguish which Type45 records exist
  -> map existing records to dynamic producers
  -> infer missing BIOS record must be static table path
  -> inspect static table guard macro
  -> trace token default to platform override
  -> confirm final generated token is 0
```

This is a useful repeatable pattern for BIOS SMBIOS bugs:

1. First decide whether the table type is globally missing or only one producer is missing.
2. Search for the displayed component names, not only the SMBIOS type number.
3. Separate static SMBIOS table generation from dynamic `SmbiosProtocol->Add()` paths.
4. For static records, always trace SDL/token override order instead of trusting the first default value seen in a package.

## How To Trace Static vs Dynamic SMBIOS Code

The practical entry point is the OS output. For Type45, `dmidecode` already exposes the SMBIOS strings:

```text
Firmware Component Name: BMC Firmware
Firmware Component Name: TPM Firmware
```

Use those visible strings as code search anchors. This is often faster than searching only for `Type45`, because one SMBIOS type can be produced by several independent modules.

Useful search pattern:

```sh
rg -n "\"BMC Firmware\"|\"TPM Firmware\"|\"BIOS Firmware\"" <bios-root>
rg -n "Type45|TYPE45|SMBIOS_TYPE_FIRMWARE_INVENTORY_INFORMATION|FIRMWARE_INVENTORY_INFO" <bios-root>
```

When the string lands in C code, inspect nearby code for these dynamic-generation signs:

```c
Type45Record->Hdr.Type = 45;
Type45Record->SmbiosType45.Hdr.Type = 45;
SmbiosProtocol->Add (..., (EFI_SMBIOS_TABLE_HEADER *) Type45Record);
```

That means the record is built at runtime by a DXE/driver path. In this case:

- `"BMC Firmware"` is a C string in `AmiIpmi2Pkg/Ipmi2/IpmiInitialize/IpmiSmbiosTypes.c`, then `UpdateSmbiosType45Record()` calls `SmbiosProtocol->Add()`.
- `"TPM Firmware"` is a C string in `AmiModulePkg/TCG2/Common/TpmSmbios/TpmSmbiosDxe.c`, then `UpdateSmbiosType45TpmRecord()` calls `SmbiosProtocol->Add()`.

When the string lands in `.dt` / `.sdl`, inspect nearby code for static-generation signs:

```text
UINT8 : 0x2D                 // Type 45
#if TYPE45_STRUCTURE
String: BIOS_FW_COMPONENT_NAME
String: BIOS_FW_VERSION
```

That means the record is generated from the static SMBIOS data table during build, not hand-built by a runtime driver. In this case:

- `"BIOS Firmware"` comes from `BIOS_FW_COMPONENT_NAME`.
- The BIOS version comes from `BIOS_FW_VERSION = $(BIOS_VERSION)`.
- The whole static Type45 block is guarded by `TYPE45_STRUCTURE`.
- `TYPE45_STRUCTURE` follows `FIRMWARE_INVENTORY_INFO`.
- A platform override turned `FIRMWARE_INVENTORY_INFO` to `0`, so this static record never entered the final SMBIOS table.

So the difference is:

```text
Dynamic Type45:
  string literal in .c
  Allocate/CopyMem/build structure
  Hdr.Type = 45
  SmbiosProtocol->Add()

Static Type45:
  raw table layout in .dt
  strings/tokens in .sdl
  guarded by SDL macros
  included or excluded at build time
```

For future SMBIOS bugs, the shortest tracing order is:

1. Copy the exact visible string from `dmidecode`.
2. Search the string first.
3. If it is in `.c`, look for `SmbiosProtocol->Add()` and the module entry/callback.
4. If it is in `.dt` / `.sdl`, trace the guard macro and all platform/project overrides.
5. Only after that search broad terms like `Type45`, because broad type searches produce too much noise.

## Secondary Suspects

If generated tokens show `TYPE45_STRUCTURE = 1`, then check these next:

1. `BIOS_VERSION` / `BIOS_TAG` may be empty or not propagated, causing a `BIOS Firmware` record with a blank or unexpected version string.
2. `DynamicUpdateBiosFirmwareInfo()` only updates firmware ID and associated handle; it does not dynamically set the BIOS version string. BIOS version is expected to already exist in the static string area as `BIOS_FW_VERSION`.
3. `DynamicUpdateBiosFirmwareInfo()` calls `FindStructureType(..., 45, 1)` and updates the first Type45 record in the static buffer. If ordering or duplicate Type45 handling changes, verify it is touching the BIOS static Type45, not a later dynamic BMC/TPM record.
4. `dmidecode 3.5` warns that SMBIOS 3.9 is newer than it fully supports. This is probably not the main issue because it still parses BMC/TPM Type45 records, but raw `dmidecode --dump-bin` can be used if output looks suspicious.
