# BIOS 工程提效与 SMBIOS 适配阶段汇报 PPT 大纲

## 审查汇总

Type9 材料适合讲“多版型显示规则怎么收敛”，不适合展开函数名、资料包、PCD 失败矩阵。

SMBIOS 材料适合讲“AI 把人工验证动作标准化执行”，但不能说真实规则自动判定和自动修代码已经完成；这部分应作为下一阶段闭环演进来讲。

AI Review 材料适合讲“旧架构能展示结果，新架构补上风险处置闭环”。样本只能说代表 case，不要夸大成全面准确率。

## 新版 PPT 大纲

建议 12 页，不做大而空的目录说明，按“结论先行 -> 三项重点工作 -> 下一步”组织。

### 1. 本阶段围绕 SMBIOS 适配和 AI 工程提效形成三项阶段成果

主结论：本阶段成果分别对应适配方案收敛、AI 工程提效和审查链路迭代。

内容：

- Type9 方案收敛。
- AI 验证和移植提效。
- AI Review 架构迭代。

建议图示：三块成果卡片，不写“汇报重点”或“快速说明”。

### 2. V5 Type9 采用 SlotNumber + Layout 表，降低多版型显示规则维护成本

主结论：Type9 的核心变化是把客户显示规则从代码分支中抽出来，由 Layout 表承接。

内容：

- 需要兼容 20+ 版型显示差异。
- 客户可见显示名以 PPT 定义为准。
- 代码不再现场拼接 Slot / Node / PCIe 显示名。

建议图示：

```text
PCIe / IIO -> SlotNumber -> Layout 表 -> DisplayName -> SMBIOS Type9
```

### 3. Type9 用特例优先匹配承接版型差异，避免继续堆代码分支

主结论：通用规则负责兜底，特例规则优先命中，冲突规则显式报错。

内容：

- 通用规则覆盖共性版型。
- Node / Multi 等特例通过更精确条件优先命中。
- 同分但显示名不同必须报错。
- 当前 AI 机型只预留扩展入口。

建议图示：通用规则 / 特例规则 / 冲突规则三列表。

### 4. AI 提效先从 SMBIOS 验证和 Type8 / Type12 porting 两个重复场景切入

主结论：AI 提效优先处理重复执行、规则整理和代码搬运工作。

内容：

- SMBIOS 验证闭环：减少上传、刷写、等待、采集、报告整理等人工操作。
- Type8 / Type12 porting：把 PPT / CSV 规则结构化后交给 opencode 辅助移植。
- 共同价值是减少人工搬运和重复确认。

建议图示：两个场景并列卡片。

### 5. SMBIOS 验证已从人工串行操作收敛为可见式 Agent 闭环

主结论：当前阶段先把人工验证流程沉淀成可执行、可观察、可复用的 Agent 闭环。

内容：

- 原人工流程包括上传 BIOS、刷写、等待 OS、采集 dmidecode、整理结果。
- Agent 接管重复执行动作，并统一输出日志和报告。
- BIOS 刷写保留 `YES_FLASH` 人工确认。

建议图示：人工流程 vs Agent 闭环对比图。

### 6. 当前 MVP 已跑通“本地 BIOS -> BMC 刷写 -> OS 采集 -> 报告”的主链路

主结论：MVP 的核心链路已收敛为本地 BIOS 输入、BMC 侧刷写、OS 侧采集和报告产出。

内容：

- 上传 BIOS 到 BMC。
- 检查并复用远端刷写脚本。
- 确认后刷写并等待 OS 恢复。
- 采集 Type 1/2/3/4/8/9/12/17/41。
- 输出合并日志和报告文件。

建议图示：

```text
Local BIOS -> BMC Upload -> Script Check -> YES_FLASH -> Flash / Reboot -> Wait OS -> Collect SMBIOS -> Report
```

### 7. AI 的直接价值是减少重复操作，并把验证过程沉淀为可追踪证据链

主结论：AI 不是替代高风险决策，而是减少重复执行和证据整理成本。

内容：

- 命令、结果、关键输出和状态可追踪。
- 失败时能定位在上传、刷写、重启、OS 恢复、采集或判定阶段。
- 当前定位是可控 Agent 流程，不是黑盒无人值守平台。

建议图示：提效矩阵。

| 人工动作 | Agent 替代方式 | 提效结果 |
|---|---|---|
| 上传 BIOS | 固定路径和 SHA256 校验 | 减少拿错文件风险 |
| 执行刷写 | 复用远端脚本并展示内容 | 减少漏步骤 |
| 等待 OS | 有界等待 SSH 恢复 | 减少人工盯守 |
| 采集 SMBIOS | 固定 Type 集合采集 | 输出稳定证据 |
| 整理报告 | 自动归档日志和报告 | 便于复盘 |

### 8. 下一阶段把 SMBIOS 规则接入 opencode，形成验证驱动修改的循环

主结论：当前闭环完成验证证据自动化，下一步把真实规则和 opencode 接起来。

内容：

- example baseline 不能作为真实验收。
- 后续接入 `smbios_requirements.json` 或 PPT / CSV 需求规则。
- FAIL 项结构化后回传 opencode 定位并修改 SMBIOS 代码。
- 修改后重新构建 BIOS，再进入刷写采集闭环。

建议图示：

```text
需求规则 / PPT / CSV -> opencode 修改 -> BIOS 构建 -> Agent 验证 -> FAIL 回传 -> 再验证
```

### 9. Type8 / Type12 porting 适合作为 AI 移植试点，用结构化规则压缩人工整理时间

主结论：Type8 / Type12 可作为 AI 从需求资料到代码移植的试点方向。

内容：

- 三份 handoff 中没有实际完成证据，因此建议按“试点方向”讲，不写成已完成。
- 路径是 PPT / CSV 需求 -> opencode 定位旧实现 -> 按 V5 命名、路径、字段要求移植。
- 预期价值是减少需求整理、代码定位和重复搬运时间。

建议图示：

```text
客户 PPT -> 脚本转 CSV -> opencode 读取 -> V5 改造 -> 编译 / 验证
```

### 10. AI Review 已从结果展示链路升级为可追踪的风险审查闭环

主结论：AI Review 当前进展不是界面优化，而是审查链路具备了风险识别、证据验证、结果展示和后台复盘能力。

内容：

- 旧链路价值是 Jenkins 触发、OpenCode 审查、Review URL 展示。
- 新链路补上下文事实包、证据补齐、风险账本、处置结果和复盘能力。
- 模型看到的风险不再黑盒消失。

建议图示：旧阶段结果展示 -> 当前风险闭环 -> 输出可解释、可追踪、可复盘。

### 11. 新架构把上下文理解前置，让规则线索经过证据验证后再形成结论

主结论：新架构先理解代码改动和上下文，再判断规则线索是否成立，避免规则直接决定结论。

内容：

- 旧链路：Jenkins -> OpenCode -> skill / 规则 -> JSON -> 网页展示。
- 新链路：Git diff -> context pack -> 覆盖检查 -> planning -> 证据补齐 -> riskLedger -> governance -> Review URL / Admin。
- 每个风险都有处置结果，便于追踪为什么展示或为什么不展示。

建议图示：左右架构对比图。

### 12. 后续重点是补真实规则、扩真实样本，把 AI 提效链路落到持续验证中

主结论：下一阶段围绕真实代码、真实规则和真实样本推进。

内容：

- Type9：落 Layout 表、匹配逻辑和典型版型验证。
- SMBIOS：接真实需求规则，推动“验证 -> 修改 -> 再验证”循环。
- Type8 / Type12：做 porting 试点，验证 PPT / CSV 到代码移植效率。
- AI Review：扩大真实 commit 和 negative 样本，固定评测报告。

建议图示：三条路线图。

```text
适配落地：Type9 Layout 表 -> 匹配逻辑 -> 典型版型验证
AI 提效闭环：规则输入 -> opencode 修改 -> Agent 验证 -> 再验证
Review 评测闭环：代表样本 -> 真实样本 -> 固定评测报告
```

- AI 提效放 6 页，作为主线。
- AI Review 放 2 到 3 页。
- Type8 / Type12 目前按“试点方向”讲，不写成已完成成果。
