# BIOS 工程提效与 SMBIOS 适配阶段汇报 PPT 详细大纲

## 制作口径

本大纲用于后续直接制作网页 PPT 或 PPTX。每页都按“标题即结论”的金字塔写法组织，页面上不出现内部讲稿式提示。

推荐风格：商务化瑞士风，白底、深灰文字、单一蓝色强调。整体克制，少用大段文字，多用流程图、对比图、矩阵表。

建议页数：12 页。Type9 控制 2 页，AI 提效作为主线 6 页，AI Review 2 页，最后 1 页收束计划。

每页制作时遵守：

- 页面标题必须是结论句，不用“背景、方案、流程、下一步”这种目录式标题。
- 每页只表达一个中心结论。
- 正文最多 3-5 个短句，能画图就不堆字。
- 代码路径、commit、压缩包名称、IP、账号密码不进主页面。
- Type8 / Type12 porting 目前按“试点方向”讲，不写成已完成成果。
- SMBIOS 自动判定和 opencode 自动修代码属于下一阶段，不写成当前已完成。
- AI Review 样本只说代表 case，不写成全面准确率。

## 1. 本阶段围绕 SMBIOS 适配和 AI 工程提效形成三项阶段成果

### 推荐版式

三卡总览页。左上放大标题，右侧或下方放三张等宽成果卡。底部只放一句收束，不放内部目录提示。

可用 Swiss 风格中的三卡/三子卡结构，视觉上保持一页只出现三个主块。

### 页面最终标题

```text
本阶段已形成三项成果：Type9 方案收敛、AI 验证提效、AI Review 闭环迭代
```

标题要占页面上半区，足够像结论，不要写成“阶段工作概览”。

### 屏幕正文文案

三张卡片：

```text
01  Type9 多版型适配
将客户 PPT 中的槽位显示差异沉淀到 Layout 表，减少后续版型适配时反复改代码分支。

02  AI 工程提效
先从 SMBIOS 验证闭环和 Type8 / Type12 porting 两个重复场景切入，减少人工执行和资料搬运。

03  AI Code Review 迭代
从“能展示审查结果”升级为“风险有证据、有处置、有复盘路径”的审查闭环。
```

底部收束句：

```text
三项工作共同指向：适配更可维护、验证更可复用、审查更可追踪。
```

### 图怎么画

三张成果卡横向排列。每张卡顶部用 `01 / 02 / 03`，中间是短标题，底部是一句价值。第一张普通白底描边，第二张淡蓝底，第三张浅灰底即可。不要使用复杂流程图。

### 讲述顺序

先说“本阶段不是单点工具，而是三条线同时推进”。再按 Type9、AI 提效、AI Review 三张卡各讲一句。最后用底部收束句过渡到 Type9。

### 不要写

- 不写内部讲稿式说明。
- 不把页面做成目录提示。
- 不放 20+、4/4 等没有上下文的数字。

## 2. V5 Type9 的关键问题是客户显示名差异多，不能继续靠代码分支维护

### 推荐版式

问题 + 方案 + 展示结果三栏页。左侧讲当前痛点，中间讲本次方案，右侧放一个最终展示示例。这样听众能先知道“为什么要改”，再知道“我们改成什么”，最后看到“最后输出长什么样”。

三栏建议比例：左侧 32%，中间 36%，右侧 32%。右侧展示示例要像结果卡，不要只写流程箭头。

### 页面最终标题

```text
V5 Type9 的关键问题是客户显示名差异多，不能继续靠代码分支维护
```

副标题可写：

```text
同一个物理槽位在不同版型下可能显示为 Slot、PCIe 或 Node-Slot，客户可见名称必须以 PPT 定义为准。
```

### 屏幕正文文案

左侧“当前问题”：

```text
当前问题
- 客户 PPT 中有 20+ 种 Type9 显示布局
- Board、Single/Multi、Node 都会影响显示名
- 继续在代码里拼 Slot / PCIe / Node，会让新增版型越来越难维护
```

中间“本次方案”：

```text
本次方案
- SlotNumber：运行时识别物理槽位
- Layout 表：维护不同版型的显示规则
- DisplayName：严格来自客户 PPT，最终进入 SMBIOS Type9
- SilkName：只用于映射校验和问题定位
```

右侧“最终展示示例”：

```text
示例
运行时槽位：SlotNumber = 7
版型条件：Multi + Node0
Layout 命中：DisplayName = Node0-Slot20
Type9 输出：SlotDesignation = Node0-Slot20
```

底部收束句：

```text
这次改动的重点不是重新枚举槽位，而是把“显示名怎么来”从代码分支迁移到可维护的规则表。
```

### 图怎么画

画成三栏结构：

```text
当前问题                 本次方案                         最终展示
20+ 显示布局      ->      SlotNumber + Layout 表     ->      Type9 SlotDesignation
代码拼显示名              PPT DisplayName                   Node0-Slot20
新增版型难维护            表驱动维护                         客户可见字段
```

中间方案栏可以放一条简化链路：

```text
PCIe/IIO -> SlotNumber -> Layout Table -> DisplayName -> SMBIOS Type9
```

右侧结果卡要突出 `SlotDesignation`，让听众知道这套方案最后影响的是客户能看到的 Type9 字段。

### 讲述顺序

先讲当前痛点：不是找不到槽位，而是同一个槽位在不同版型下显示名不同。再讲本次方案：代码负责拿 SlotNumber，Layout 表负责给 DisplayName。最后指到右侧展示：最终进入 Type9 的是客户 PPT 里定义的显示名。

### 不要写

- 不讲 `FindBestSlotDesc()`、`BuildType9FromRootBridge()` 等函数名。
- 不放完整 CSV 字段列表。
- 不讲 V3 AI 逻辑细节。
- 不把 SilkName 写成运行时主 key。

## 3. Layout 表通过“通用兜底 + 特例优先”选出最终 Type9 显示名

### 推荐版式

输入 -> 规则匹配 -> 输出展示页。左侧放运行时输入，中间放 Layout 表命中示例，右侧放最终 Type9 输出。第 2 页讲“为什么需要表”，第 3 页讲“表怎么选出正确显示名”。

### 页面最终标题

```text
Layout 表通过“通用兜底 + 特例优先”选出最终 Type9 显示名
```

### 屏幕正文文案

左侧“运行时输入”：

```text
运行时输入
Board = C2-8CXL
HostMode = Multi
Node = Node0
SlotNumber = 7
```

中间“Layout 表命中示例”：

```text
候选规则
1. Any + NoNode + SlotNumber 7 -> Slot20
2. Multi + Node0 + SlotNumber 7 -> Node0-Slot20
3. Multi + Node1 + SlotNumber 7 -> Node1-Slot20

当前输入命中规则 1 和规则 2，规则 2 条件更精确，因此选中 Node0-Slot20。
```

右侧“最终输出”：

```text
SMBIOS Type9
SlotDesignation = Node0-Slot20

客户看到的显示名来自 PPT，不由代码现场拼接。
```

底部风险控制：

```text
同分但 DisplayName 不同，直接报规则冲突；异常场景下可以少显示，不能显示错。
```

### 图怎么画

画成三段式：

```text
运行时上下文
Board / HostMode / Node / SlotNumber
        ↓
Layout 表候选规则
通用规则、Node0 特例、Node1 特例
        ↓
Type9 输出
SlotDesignation = Node0-Slot20
```

中间 Layout 表不要做成普通 Excel。建议做 3 行规则卡，第二行用蓝色强调“SELECTED”。底部用一个橙色小条写“冲突规则 -> ERROR”。

### 讲述顺序

先接上页：Layout 表已经承接显示规则，这页说明它怎么选。然后按左中右讲：运行时拿到 Board/HostMode/Node/SlotNumber，中间找到候选规则，条件更精确的特例胜出，右侧输出 Type9 显示名。最后补一句：如果规则同分但显示名不同，就报错，不让错误显示名静默出货。

### 不要写

- 不写“打分算法”作为主标题，领导容易觉得是算法炫技。
- 不放 PCD 失败矩阵。
- 不在这一页展开 AI Provider，只说预留扩展能力即可。
- 不只放“通用规则 / 特例规则 / 冲突规则”三列表，否则和上一页关联不够。

双场景并列页。左半是 SMBIOS 验证闭环，右半是 Type8 / Type12 porting。中间用一条竖线或箭头连接到底部共同价值。

### 页面最终标题

```text
AI 提效优先处理两类重复工作：验证执行和规则移植
```

### 屏幕正文文案

左侧卡片：

```text
场景一：SMBIOS 验证闭环
上传 BIOS、执行刷写、等待 OS、采集 dmidecode、归档报告。

价值：减少人工重复操作，让验证过程可追踪。
```

右侧卡片：

底部共同价值：

```text
AI 先处理重复执行和资料整理，人工保留规则确认、风险授权和最终复核。
```

```text

PPT/CSV -> opencode -> V5 代码 -> 验证
```

### 不要写

- 不写成“AI 已经完全自动修代码”。
- Type8 / Type12 不写成已完成，只写试点方向。
- 不要把 SMBIOS 验证和 AI Review 混成一个场景。

## 5. SMBIOS 验证已从人工串行操作收敛为可见式 Agent 闭环

### 推荐版式

Before / After 对比页。左侧是人工流程，右侧是 Agent 流程，中间用分割线。页面下方放一句风险控制说明。

### 页面最终标题

```text
SMBIOS 验证从人工串行操作变成可见式 Agent 流程
```

### 屏幕正文文案

左侧“人工流程”：

```text
1. 手动上传 BIOS
2. SSH 到 BMC 执行刷写
3. 人工等待刷写和重启
4. 登录 OS 执行 dmidecode
5. 手动整理日志和结论
```

右侧“Agent 流程”：

```text
1. 输入本地 BIOS 路径
2. 上传并校验文件大小和 SHA256
3. 检查远端刷写脚本
4. YES_FLASH 后执行刷写
5. 等待 OS 恢复并采集 SMBIOS
6. 生成日志和报告
```

底部说明：

```text
刷写仍保留人工确认，提效不等于放弃风险控制。
```

### 图怎么画

左边用灰色 5 步纵向流程，右边用蓝色 6 步纵向流程。右侧每一步旁边可以放 `PASS / BLOCKED / NEXT` 的小标签，体现可见式反馈。

### 讲述顺序

先讲“原来最耗时间的不是单条命令，而是跨 BMC、OS、本地反复切换”。再讲 Agent 把这些步骤固定下来。最后强调 YES_FLASH 是安全闸门。

### 不要写

- 不写具体 IP、账号、密码。
- 不讲 500 次重启脚本。
- 不提 Jenkins 下载失败细节。

## 6. 当前 MVP 已跑通“本地 BIOS -> BMC 刷写 -> OS 采集 -> 报告”的主链路

### 推荐版式

横向流程页。中间放 8 个节点，节点之间用箭头连接。每个节点只放短动词，不写长段落。

### 页面最终标题

```text
当前 MVP 主链路已收敛为本地 BIOS、BMC 刷写、OS 采集和报告产出
```

### 屏幕正文文案

流程节点：

```text
01 本地 BIOS
输入本地 bin 路径

02 上传 BMC
上传到 /tmp 并校验大小

03 检查脚本
复用远端刷写脚本

04 风险确认
等待 YES_FLASH

05 执行刷写
记录 flash 输出

06 等待 OS
有界等待 SSH 恢复

07 采集 SMBIOS
采集 Type 1/2/3/4/8/9/12/17/41

08 生成报告
输出日志、summary、json
```

底部补充：

```text
当前重点是稳定采集证据；真实 PASS / FAIL 规则下一阶段接入。
```

### 图怎么画

横向 8 步流程，最好分两行：

第一行：`本地 BIOS -> 上传 BMC -> 检查脚本 -> YES_FLASH`  
第二行：`执行刷写 -> 等待 OS -> 采集 SMBIOS -> 生成报告`

`YES_FLASH` 节点用橙色或描边强调，表示风险闸门。

### 讲述顺序

按流程从左到右讲，不展开每个命令。讲到 `YES_FLASH` 时强调“风险动作没有交给模型自行决定”。讲到报告时说明“这是下一阶段自动判定和修代码的输入”。

### 不要写

- 不在页面上写 `root/root`、`root/111111`。
- 不说“真实 SMBIOS 判定已完成”。
- 不把 example expected 的 FAIL 放成成果。

## 7. AI 的直接价值是减少重复操作，并把验证过程沉淀为可追踪证据链

### 推荐版式

提效矩阵页。左侧放一句结论，右侧放 5 行矩阵：人工动作、Agent 替代方式、提效结果。矩阵要比普通表格更像“价值说明”，每行只保留关键短语。

### 页面最终标题

```text
AI 的直接价值是减少重复操作，并把验证过程沉淀为可追踪证据链
```

### 屏幕正文文案

矩阵内容：

| 人工动作 | Agent 替代方式 | 提效结果 |
|---|---|---|
| 上传 BIOS | 固定路径上传并校验 SHA256 | 减少拿错文件风险 |
| 执行刷写 | 检查脚本后统一执行 | 减少漏步骤 |
| 等待 OS | 有界等待 SSH 恢复 | 减少人工盯守 |
| 采集 SMBIOS | 固定 Type 集合采集 | 输出稳定证据 |
| 整理报告 | 自动归档日志和报告 | 便于复盘和后续判定 |

页面底部：

```text
每一步保留命令、关键输出和状态，失败时能定位在具体环节。
```

### 图怎么画

用三列表格，但不要做成密密麻麻的 Excel。每行左侧可加小图标：上传、终端、时钟、列表、报告。第三列用蓝色小标签突出提效结果。

### 讲述顺序

先讲“提效不是一句口号，要拆到具体动作”。然后按矩阵讲 2-3 行代表即可，不必逐行念完。最后强调证据链对后续自动判定和自动修改很重要。

### 不要写

- 不写“AI 替代审核”。
- 不写“无人值守刷写”。
- 不放运行日志原文。

## 8. 下一阶段把 SMBIOS 规则接入 opencode，形成验证驱动修改的循环

```text
下一阶段把验证结果变成 opencode 修改代码的输入
```

### 屏幕正文文案

```text
需求规则
PPT / CSV / smbios_requirements.json
        ↓
```

### 讲述顺序

### 不要写

- 不写“已经自动修代码”。
- 不写“example baseline 判定通过/失败”。
- 不把 `smbios_requirements.json` 写成已实现。

## 9. Type8 / Type12 porting 适合作为 AI 移植试点，用结构化规则压缩人工整理时间

### 推荐版式

流程 + 预期收益页。上方放一句结论，中间放从客户 PPT 到 V5 代码的 5 步流程，底部放“人工方式 / AI 辅助方式”的耗时对比，但用“预期”口径。

## 制作检查清单

制作 PPT 前逐页检查：

- 每页标题是否是结论句。
- 页面是否只有一个中心意思。
- 是否至少 8 页有主图、流程、矩阵或对比结构。
- 是否避免了内部讲稿式提示。
- Type8 / Type12 是否写成试点方向，而不是已完成成果。
- SMBIOS 自动判定和 opencode 自动修代码是否写成下一阶段，而不是当前成果。
- AI Review 评测是否只说代表 case，不夸大全面准确率。
- 是否没有出现 IP、账号、密码、commit 考古和大量文件路径。
