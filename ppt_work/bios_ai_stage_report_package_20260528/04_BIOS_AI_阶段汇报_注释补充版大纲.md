## Slide 6：Type9 重构

**页面结论**：Type9 多版型适配从“重复改代码”转为“维护显示规则”。

页面内容：
- 改造前：板型、部署模式、节点配置散落在代码分支中。
- 改造前：新增版型需要继续改主流程，复用性弱、维护成本高。
- 改造后：差异集中到显示规则。
- 改造后：主流程只负责采集、选择、生成。
- 核心链路：底层槽位信息 → 统一槽位编号 → 显示规则 → 客户可见槽位名。
- 价值：新增版型优先补显示规则，不反复改核心流程，提升代码复用性，减少维护时间；同时降低现场拼接显示名带来的误显示风险。

## Slide 8：SMBIOS 自动验证

**页面结论**：SMBIOS 验证从“人工多环境切换”沉淀为基于 opencode skills 的可追踪自动化流程。

页面内容：
- 旧方式：人工准备 BIOS、手动切换测试环境、手动刷写、手动执行 dmidecode、手动整理日志和报告。
- 旧方式问题：操作链路长，容易漏步骤；刷写、采集、比对结果分散，不利于复盘。
- 新方式：通过 opencode + skills 固化验证动作，将上传、校验、刷写、采集、报告生成串成可重复流程。
- 新方式流程：BIOS 上传 → SHA256 校验 → 远程刷写 → 人工确认 → 系统侧采集 → 报告生成。

覆盖范围：
- 9 类 SMBIOS 数据采集。
- 保留刷写前人工确认点，避免高危操作完全无人介入。
- 输出原始日志和结构化报告，方便后续 Agent 读取现象并辅助定位。

证据位置：
- 右侧放验证日志截图和 dmidecode 输出截图。

版式建议：
- 左侧放“旧方式 / 新方式”对比。

## Slide 10：AI Review 上下文增强

**页面结论**：AI Review 新增上下文完整性检查，降低模型只看局部 diff 的漏判风险。

页面内容：
- 旧方式：模型主要读取 diff 和固定规则；如果评论较少，不容易判断是没有风险，还是上下文不够。
- 当前方式：代码变更 → 上下文收集 → 完整性检查 → 关联分析 → 证据补充 → AI Review。
- 能力卡片 1：上下文材料，补充变更文件、函数窗口、EDK 元数据和上报链路。
- 能力卡片 2：完整性检查，记录模块、配置、调用链和上报路径是否足够。
- 能力卡片 3：过程留痕，输出上下文覆盖记录，管理员可回看缺什么、补了什么。
- 数据支撑：6 个真实提交回放成功运行；4 / 4 历史问题反向样本在中间过程触达预期风险；2 个 negative 正常改动未进入正式问题。

## Slide 12：下一步计划

**页面结论**：下一步聚焦 SMBIOS 可观测问题，沉淀开发生成和测试修复两条 Agent 流程。

开发阶段工作流：
- 需求 / 客户规则 → 规格文档 → opencode skills → AI 辅助生成代码 → 自动化验证 → 人工验收 → 上库。
- 适用：规则清晰、字段边界明确、输出可比对的 SMBIOS 模块。

测试问题工作流：
- 自动验证发现差异 → Agent 读取日志和代码 → 定位上报路径 → 输出修复建议 → 工程师确认 → 复测回归。
- 适用：字段缺失、显示异常、客户可见信息不一致、配置开关导致上报缺失。

边界：
- 不做泛化 BIOS Bug 自动修复；启动失败、硬件链路异常、时序不稳定、偶发平台问题不纳入本轮闭环。

验收：
- 选取 2-3 类 SMBIOS 显示 / 字段问题，保留验证报告、定位记录、修复建议、人工确认和复测结果。
